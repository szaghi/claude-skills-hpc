# Chapter 1: Overview of the OpenMP API

## Core Idea
OpenMP is a **user-directed**, portable parallel-programming model extending C/C++/Fortran with directives, runtime routines, environment variables, and tool interfaces (OMPT/OMPD). The programmer is responsible for correctness — the implementation does **not** check for data races, dependences, or deadlocks.

## Frameworks Introduced
- **Execution model — the entity hierarchy**:
  - **device** → executes one or more **contention groups**.
  - **contention group** = a set of tasks run by an **OpenMP thread pool** (a fixed set of threads; threads cannot migrate between contention groups).
  - **team** = the threads executing a `parallel` region (primary thread + team-worker threads).
  - **league** = the set of teams created by a `teams` construct (each team has its own contention group).
  - **thread** (logical) → mapped to a native thread (mapping may change across subtasks).
- **Construct / region / task model**:
  - **construct** = an executable directive + its associated base-language code.
  - **region** = all code encountered during execution of a construct/routine (dynamic, includes callees — implicit or explicit, and nestable).
  - **implicit task** (one per thread in a team, **tied** — only its thread executes it) vs **explicit task** (generated by `task`/`taskloop`, may be deferred/stolen).
  - **task scheduling point**: where a thread may suspend the current task to run another; subtasks span consecutive scheduling points.
- **Four construct families** the directives add: SPMD/parallelism (`parallel`, `teams`), **work-distribution** (worksharing-loop, `sections`, `single`, `distribute`), **tasking** (`task`, `taskloop`), **device** (`target`), plus **synchronization** and SIMD (`simd`).

## Key Concepts
- **fork–join + barriers**: `parallel` forks a team, implicit barrier at the end; worksharing regions also barrier unless `nowait`. Team-executed regions are encountered in the same order by all team threads.
- **target offload**: a task hitting `target` generates a **target task**; an initial thread runs the `target` region on the device (or host if device unavailable / `if(false)`).
- **teams concurrency is unspecified** — relying on concurrent execution of teams for synchronization may **deadlock**.
- **cancellation** (`cancel`): alters control flow; tasks check at cancellation points and jump to region end.
- **Tool interfaces**: **OMPT** (first-party, in-process profiling) and **OMPD** (third-party, out-of-process debugging) — ch18.
- **Compliance**: the implementation need not detect non-conforming programs; conformance is the programmer's burden.

## Mental Models
- **Three orthogonal parallelism axes**: threads (fork–join `parallel`), tasks (deferred work units), and devices (`target` offload) — plus SIMD within a thread. Modern OpenMP code composes all of them.
- **"Region" is dynamic, "construct" is static** (same as OpenACC): reason about data and synchronization over the region, including called procedures.
- **The implementation trusts you**: no race/deadlock checking. Correctness = your explicit data-sharing, synchronization, and dependence specification.

## Anti-patterns
- **Assuming the compiler catches races/deadlocks**: it does not — you must specify `shared`/`private`/`reduction` and synchronization correctly.
- **Relying on teams executing concurrently**: unspecified; cross-team synchronization can deadlock (use it only within a team).
- **Expecting bit-reproducible numerics under threading**: concurrent reassociation of operations changes results.
- **Confusing implicit vs explicit tasks**: implicit tasks are tied to their team thread; explicit tasks may be deferred or stolen (untied).

## Key Takeaways
1. OpenMP is user-directed: you own correctness; the implementation checks nothing.
2. Entity hierarchy: device ⊃ contention group (thread pool) ⊃ team; `teams` makes a league of teams.
3. Three parallelism axes — threads (`parallel`), tasks (`task`), devices (`target`) — plus SIMD; modern code composes them.
4. `parallel` = fork-join with an end barrier; worksharing regions barrier unless `nowait`.
5. OMPT (profiling) + OMPD (debugging) are the standardized tool interfaces (ch18).
