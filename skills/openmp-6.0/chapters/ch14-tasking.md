# Chapter 14: Tasking Constructs

## Core Idea
**Explicit tasks** — deferrable units of work with dependences — for irregular, recursive, and dynamic parallelism: `task`, `taskloop`, the new 6.0 **`taskgraph`** (record/replay a task graph), plus synchronization (`taskwait`, `taskgroup`) and the `depend` clause.

> ⚠ **Errata (Nov 2025)** applied below at §14.3 (`taskgraph`), §14.3.2 (`graph_reset`), §14.6 (`replayable`).

## Frameworks Introduced
- **`task`** (§14.1): generate an explicit task. Clauses: `depend(dep-type: list)`, `if`, `final`, `mergeable`, `priority`, `untied`, `detach(event)`, `affinity`, `in_reduction`, data-sharing clauses, and 6.0 **`replayable`**, **`threadset`**, **`transparent`**. A task scheduling point is created at the construct.
- **`taskloop`** (§14.2): partition loop iterations into explicit tasks. Implicitly wrapped in a `taskgroup` unless `nogroup`. Clauses: `grainsize`/`num_tasks`, `collapse`, `reduction`, `priority`, plus `task_iteration` subsidiary directives. 6.0 adds `threadset`, `transparent`, `replayable`.
- **`taskgraph`** (§14.3, **6.0**): record a region's task graph once, then **replay** it on subsequent encounters — amortizing task-creation overhead for repeated, stable task patterns (e.g. iterative solvers). `graph_reset` (§14.3.2) forces re-recording.
- **`depend(dep-type: list)`**: data-flow dependences — `in`, `out`, `inout`, `mutexinoutset`, `inoutset`, `depobj`. Builds the task DAG; `iterator(...)` generates a dependence per element.
- **`taskwait`** (wait for child tasks, optionally `depend`), **`taskgroup`** (wait for all descendants; supports task reductions), **`taskyield`** (task scheduling point).

## Errata corrections (Nov 2025)
- **§14.3 (`taskgraph`)**: added restriction — *if a task generated by one construct in a taskgraph region is an antecedent of a task generated by another construct in the region, **either both constructs must be `replayable` or both must be not**.* (Can't mix replayable/non-replayable across a dependence.)
- **§14.3.2 (`graph_reset` clause)**: the `default` property is changed to **`optional`**.
- **§14.6 (`replayable` clause)**: the `default` property is changed to **`unique`**.

## Key Concepts
- **`depend` builds the DAG**: `out`/`inout` on a producer, `in` on a consumer creates a happens-before edge — the runtime schedules respecting it. This replaces manual barriers for fine-grained pipelines.
- **`taskgraph` (6.0)** targets the overhead of *re-creating the same task graph every iteration*: record once, replay cheaply. The errata pins down replayability consistency across dependent constructs.
- **`taskloop` auto-`taskgroup`**: it waits for its tasks unless `nogroup` — so a following statement sees completed work.
- **`final(cond)` + `mergeable`**: cut task-creation overhead at recursion leaves (final tasks generate no further deferred tasks).
- **task reductions**: `reduction` on `taskgroup`/`taskloop` + `in_reduction` on participating tasks.

## Code Examples
```c
// dependence-driven pipeline: the runtime orders by data flow, no barriers
#pragma omp parallel
#pragma omp single
{
  #pragma omp task depend(out: x)        x = produce();
  #pragma omp task depend(in: x, out: y) y = stage2(x);
  #pragma omp task depend(in: y)         consume(y);
}   // single's implicit barrier waits for all

// recursive tasking with a cutoff
void quicksort(int *a, int lo, int hi) {
  if (hi - lo < THRESHOLD) { serial_sort(a, lo, hi); return; }
  int p = partition(a, lo, hi);
  #pragma omp task final(hi-lo < BIG) firstprivate(lo,p) quicksort(a, lo, p-1);
  #pragma omp task final(hi-lo < BIG) firstprivate(p,hi) quicksort(a, p+1, hi);
  #pragma omp taskwait
}
```
- **Demonstrates**: `depend`-built DAG replacing barriers, and recursive `task` with `final` cutoff + `taskwait`.

## Anti-patterns
- **Generating tasks outside `single`/`masked`**: every team thread generating the same tasks → duplication; generate from one thread.
- **Manual flags/barriers instead of `depend`**: fragile — let the `depend` DAG schedule.
- **No `final`/cutoff in recursive tasking**: unbounded fine-grained tasks → creation overhead dominates.
- **Mixing replayable/non-replayable across a `taskgraph` dependence** (post-errata): now explicitly forbidden.
- **Assuming `taskloop` doesn't synchronize**: it implies a `taskgroup` unless `nogroup`.

## Key Takeaways
1. `task` + `depend(in/out/inout/...)` builds a data-flow DAG the runtime schedules — replace barriers with dependences.
2. **`taskgraph` (6.0)** records a task graph once and replays it — for repeated stable patterns (iterative solvers).
3. **Errata**: `taskgraph` antecedent tasks must agree on replayability; `graph_reset` default→optional; `replayable` default→unique.
4. Generate tasks from one thread (`single`/`masked`); use `final`/`mergeable` cutoffs in recursion.
5. `taskloop` implies a `taskgroup` unless `nogroup`; task reductions via `taskgroup`/`in_reduction`.
