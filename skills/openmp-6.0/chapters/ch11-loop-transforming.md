# Chapter 11: Loop-Transforming Constructs

## Core Idea
Directives that **reshape a loop nest before parallelization** — `tile`, `unroll`, `reverse`, `interchange`, `fuse`, `split`, `stripe` — composed via the `apply` clause. They transform the canonical loop nest (ch6) so a subsequent worksharing/`simd` construct sees a better-shaped iteration space.

## Frameworks Introduced
- **`tile sizes(s1, s2, ...)`**: block a loop nest into tiles for cache locality; produces an outer (inter-tile) and inner (intra-tile) loop nest.
- **`unroll [full | partial(n)]`**: unroll a loop fully or by factor n.
- **`reverse`**: iterate a loop backward (logical iteration order reversed).
- **`interchange permutation(...)`**: swap loop order in a nest (for locality/vectorization).
- **`fuse` / `split` / `stripe`**: merge adjacent loops / split one loop / strip-mine.
- **`apply` clause** (§11.1): compose transformations — apply listed loop-transforming directives to the loops generated by this one (transformations chain, e.g. tile then unroll the inner tile loop). List items needn't be unique; default loop-modifier argument is 1.

## Key Concepts
- **Transformations happen *before* worksharing**: `#pragma omp tile` then `#pragma omp parallel for` parallelizes the *tiled* loops. Compose deliberately.
- **`apply`** lets one construct drive a pipeline of transforms on its generated loops (e.g. `tile` whose inner loops are then `unroll`ed) — the 6.0 way to express multi-stage loop optimization declaratively.
- These are **hints with defined semantics**: the transformed loop nest is well-specified, unlike a compiler's opaque auto-transform.
- The result must still be a canonical loop nest for any subsequent parallelization.

## Code Examples
```c
// tile for cache, then parallelize the inter-tile loops
#pragma omp tile sizes(64, 64)
#pragma omp parallel for collapse(2)
for (int i = 0; i < n; ++i)
  for (int j = 0; j < m; ++j) a[i][j] = b[i][j] + c[i][j];

// compose via apply: tile, then unroll the intra-tile loop
#pragma omp tile sizes(32) apply(intratile: unroll partial(4))
for (int i = 0; i < n; ++i) work(i);
```
- **Demonstrates**: `tile` feeding a `parallel for collapse`, and the `apply` clause chaining `unroll` onto the intra-tile loop generated by `tile`.

## Anti-patterns
- **Applying a transform to a non-canonical loop**: it must meet ch6's canonical-loop rules first.
- **Assuming order doesn't matter**: tiling then parallelizing differs from parallelizing then tiling — the construct order defines the transformation pipeline.
- **`fuse` on loops with differing trip counts / dependences**: restricted; verify the loops are fusion-legal.
- **Over-tiling**: tile sizes unrelated to cache/vector width can hurt — tie sizes to the target's cache line / L1.

## Key Takeaways
1. `tile`, `unroll`, `reverse`, `interchange`, `fuse`, `split`, `stripe` reshape the loop nest with *defined* semantics.
2. Transformations apply **before** a subsequent worksharing/`simd` construct sees the loops.
3. The **`apply` clause** composes transforms on generated loops (e.g. tile → unroll intra-tile) — declarative multi-stage optimization.
4. The transformed nest must remain canonical for parallelization.
5. Construct order = transformation pipeline order.
